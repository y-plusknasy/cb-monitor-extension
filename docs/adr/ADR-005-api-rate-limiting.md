# ADR-005: Cloud Functions API レート制限

## ステータス

採用（Accepted）

## 決定

案 A（Firestore ベースのスライディングウィンドウ・デバイス単位）を採用。加えて以下も実装:

- **日付バリデーション**: 未来日・31 日超過去を拒否
- **strict バリデーション**: Zod `.strict()` で未定義プロパティを拒否
- **入力サニタイズ**: `appName` は英数字・ドット・ハイフン・アンダースコアのみ許可、`deviceName` は制御文字・バックスラッシュを禁止

## 背景

Chrome Extension はペアリング済み `deviceId` を用いて Cloud Functions の `usageLogs` エンドポイントにリクエストを送信する。`deviceId` は Extension ユーザーが DevTools 等から容易に取得できるため、悪意あるユーザーが自身の `deviceId` を利用して API エンドポイントに DoS 攻撃を仕掛けるリスクがある。

### Extension の正常な送信パターン

- **定期送信**: 60 秒ごとに日次バッファを送信
- **割り込み送信**: フォーカス喪失・タブクローズ時
- **バースト**: 1 回の送信で、追跡中のアプリ数分の POST を `for...of await` で逐次送信（例: chrome + youtube.com + duolingo.com → 3 リクエスト）
- 各リクエストはネットワーク往復を含め数百 ms 間隔

### 防御すべき攻撃

- 単一 `deviceId` からの大量リクエスト（DoS）
- Firestore への書き込みコスト増大
- Cloud Functions のインスタンス浪費

## 選択肢

### 案 A: Firestore ベースのスライディングウィンドウ（デバイス単位）

`devices` ドキュメントに `requestCount` / `requestWindowStart` フィールドを持たせ、一定期間内のリクエスト数を制限する。

- **ウィンドウ**: 60 秒
- **上限**: 30 リクエスト/分 (= 0.5 req/sec 平均、バースト許容)
- **メリット**: 既に `devices` ドキュメントを読み込んでいるため追加の読み取りコストなし。グローバルに一貫した制限。
- **デメリット**: `requestCount` の更新に write コスト発生（ただし `lastSeenAt` 更新に相乗り可能）。厳密な同時リクエスト対応にはトランザクションが必要。

### 案 B: 最小リクエスト間隔（`lastSeenAt` ベース）

既に更新している `lastSeenAt` を利用し、前回リクエストからの経過時間が閾値未満なら 429 で拒否する。

- **間隔**: 400ms (= 最大 2.5 req/sec)
- **メリット**: 実装が最もシンプル。追加フィールド不要。
- **デメリット**: Extension のバースト送信（複数アプリ分の逐次送信）が拒否される可能性が高い。ネットワーク往復を含む各リクエスト間隔が 400ms を下回る場合がある。

### 案 C: 案 A + 案 B のハイブリッド

短期バースト（数百 ms 間隔の連続リクエスト）は案 A のウィンドウで許容しつつ、分あたりの上限で長期的な濫用を防ぐ。

## 判断をお願いしたい点

（すべて採用済み）

1. **案 A（分単位ウィンドウ）** → 採用。60 秒ウィンドウ・上限 30 req/分。
2. **429 受信時の Extension 側** → 次のバッチ送信までスキップ（ローカルバッファ保持）。
3. **日付バリデーション** → 採用。未来日・31 日超過去を拒否。
