# ADR-003: chrome.storage.local クリア時の deviceId 消失復旧方針

> **ステータス**: Accepted  
> **作成日**: 2026-03-01  
> **決定日**: 2026-03-01  
> **決定者**: オーナー

---

## 背景 (Context)

現在の設計では、Chrome Extension の `deviceId` は `chrome.storage.local` に永続保存される。しかし、以下の操作により `chrome.storage.local` が全クリアされる可能性がある:

- Chrome ブラウザの「閲覧データの削除」（Cookie / サイトデータのクリア）
- 拡張機能のアンインストール→再インストール
- Chromebook の Powerwash（初期化）
- Chrome OS のユーザープロファイルリセット

`chrome.storage.local` がクリアされると、deviceId が消失し、拡張機能は新規インストールと同じ状態になる。新しい deviceId が生成されるが、Firebase 側には旧 deviceId が登録されたままとなり、以下の問題が発生する:

1. **保護者側**: 旧 deviceId からのデータ送信が途絶えるが、それが意図的（デバイス廃止）なのか事故（キャッシュクリア）なのか判別できない
2. **子供側**: 新 deviceId は未ペアリング状態となり、利用ログが Firebase に送信されなくなる（ペアリング済みチェックにより送信がブロックされる）
3. **データの断絶**: 旧 deviceId のデータと新 deviceId のデータが別デバイスとして蓄積される

### 他アプリの事例

| アプリ / サービス                       | ID 消失時の対策                                                                                       |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Google Family Link**                  | デバイス自体が Google アカウントに紐付くため、Chrome ストレージに依存しない。デバイス管理は OS レベル |
| **Qustodio (ペアレンタルコントロール)** | デバイスごとにアカウントログインする設計。デバイスID ではなくアカウント認証で紐付け                   |
| **Screen Time (Apple)**                 | Apple ID と端末が紐付いており、ローカルストレージに依存しない                                         |
| **Bark / Net Nanny**                    | インストール時にアカウント認証を行う設計。ストレージクリアされても再ログインで復旧可能                |

**共通パターン**: 多くのペアレンタルコントロールアプリは「ローカル生成の匿名 ID」ではなく「アカウント認証」に依存しており、ストレージクリアの影響を受けにくい。ただし、それは Chrome Extension の制約（子供にアカウント認証させたくない設計方針）とトレードオフになる。

---

## 検討した選択肢

### 選択肢 A: 無操作検知 + 保護者通知

Firebase 側で各デバイスの最終データ送信日時を監視し、一定期間（例: 3 日間）データ送信がない場合に保護者のモバイルアプリにプッシュ通知を送る。

**仕組み:**

- `devices` コレクションに `lastSeenAt` フィールドを追加
- `usageLogs` 受信時に `lastSeenAt` を更新
- Scheduled Function で日次チェック: `lastSeenAt` が閾値を超えたデバイスを検出
- 保護者のモバイルアプリに通知（FCM）: 「〇〇のデバイスから3日間データが届いていません。再設定が必要かもしれません」
- 保護者が OTP を再発行し、子供に入力させて再ペアリング

**メリット:**

- 実装がシンプル（既存のデータフローに乗る）
- deviceId 消失以外の問題（ネットワーク障害、デバイス故障等）もカバーできる汎用的なアラート
- 子供側の拡張機能に変更不要

**デメリット:**

- 検知まで最大 3 日のラグがある（利用頻度が高い場合はもっと早く気づくが）
- false positive（長期休暇中にデバイス使用しない場合等）が発生しうる
- 再ペアリングには保護者の手動操作が必要

### 選択肢 B: chrome.storage.sync によるバックアップ

`chrome.storage.sync` は Chrome アカウント（Google アカウント）に紐付くストレージで、デバイス間で同期される。deviceId と pairingStatus を `chrome.storage.sync` にもバックアップしておけば、`chrome.storage.local` がクリアされても復元可能。

**仕組み:**

- ペアリング成功時: `chrome.storage.sync` にも `{deviceId, pairingStatus}` を保存
- Extension 初期化時: `chrome.storage.local` に deviceId がなければ `chrome.storage.sync` から復元
- Powerwash 後も、同じ Google アカウントで Chrome にログインすれば deviceId が復元される

**メリット:**

- 保護者の介入なしで自動復旧できる
- deviceId の連続性が保たれる（データの断絶がない）
- Chromebook Powerwash 後も復元可能（同アカウントでサインインする限り）

**デメリット:**

- `chrome.storage.sync` には容量制限がある（QUOTA_BYTES_PER_ITEM = 8,192 バイト、QUOTA_BYTES = 102,400 バイト）。deviceId + pairingStatus 程度なら問題ない
- 子供が別の Google アカウントでサインインした場合は復元不可
- `chrome.storage.sync` 自体がクリアされるケースもゼロではない（が、`chrome.storage.local` よりは resilient）
- Chromebook の管理設定で sync が無効化されている場合は使えない

### 選択肢 C: Extension ID + デバイス特徴によるサーバーサイド再紐付け

Extension インストール時に、deviceId に加えてデバイスの指紋情報（画面解像度、OS情報、タイムゾーン等の組み合わせ）をサーバーに送信し、deviceId 消失後に同一デバイスを推定する。

**メリット:**

- 完全自動で復旧できる可能性がある

**デメリット:**

- デバイスフィンガープリントはプライバシー上問題がある
- Chrome Extension からアクセスできるデバイス情報は限定的
- 同一モデルの Chromebook が複数ある場合（学校配布等）は区別困難
- 精度が保証できない

### 選択肢 D: 選択肢 A + B の併用

`chrome.storage.sync` バックアップを第一復旧手段とし、それでも復旧できない場合（sync 無効、別アカウント等）に備えて無操作検知 + 保護者通知をフォールバックとして実装する。

**メリット:**

- 大半のケース（Powerwash、キャッシュクリア）は自動復旧
- 自動復旧できないエッジケースも保護者通知でカバー
- 最も堅牢な対策

**デメリット:**

- 実装コストが最も高い（両方の仕組みを構築する必要がある）
- chrome.storage.sync の容量制限・利用可否を考慮するコードが必要

---

## 推奨案

**選択肢 D（A + B 併用）** を推奨するが、段階的に実装可能:

1. **S02 (今回)**: `chrome.storage.sync` バックアップ（選択肢 B）のみ実装。最小コストで大半のケースをカバー
2. **S04 以降**: 無操作検知 + 保護者通知（選択肢 A）を追加。これはモバイルアプリ（S03）の FCM 通知基盤が前提

ただし、方針決定はオーナー判断を仰ぐ。

## 決定事項

**選択肢 D（A + B 併用）** を採用する。

### 実装スケジュール

| 項目                                                                                              | スプリント     | 備考                                     |
| ------------------------------------------------------------------------------------------------- | -------------- | ---------------------------------------- |
| データモデル設計 (`devices.lastSeenAt`, `devices.syncAvailable`, `users.inactivityThresholdDays`) | S02（今回）    | Firestore スキーマ + CRUD                |
| `chrome.storage.sync` バックアップ・リストア                                                      | S02（今回）    | デバイスフィンガープリントによる端末識別 |
| `usageLogs` 受信時の `lastSeenAt` 更新                                                            | S02（今回）    | 既存フローへの追加                       |
| `syncAvailable` フラグの検知・送信                                                                | S02（今回）    | 登録時にExtensionが判定                  |
| 無操作検知の閾値を保護者が設定するUI                                                              | 後続スプリント | モバイルアプリ実装後                     |
| 無操作検知 Scheduled Function + FCM 通知                                                          | 後続スプリント | モバイルアプリの FCM 基盤が前提          |
| `syncAvailable = false` デバイスへの保護者アプリ上アラート表示                                    | 後続スプリント | モバイルアプリ実装後                     |

### 無操作検知の閾値

- デフォルト: **6 日間**（平日に全くデバイスを使わないケースを考慮）
- 保護者がモバイルアプリから任意に変更可能にする（後続スプリント）
- `users` コレクションに `inactivityThresholdDays` フィールドを追加

### chrome.storage.sync の設計方針

- `chrome.storage.sync` はアカウント単位の同期ストレージであるため、同一アカウントで複数デバイスを使用するケースに対応が必要
- デバイスフィンガープリント（`navigator.userAgent` + `navigator.hardwareConcurrency` 等）をキーとして端末ごとにバックアップを格納
- sync が管理ポリシーで無効化されている場合は、`devices` コレクションの `syncAvailable = false` を記録し、保護者アプリでアラートを表示する（アプリ側表示は後続スプリント）
